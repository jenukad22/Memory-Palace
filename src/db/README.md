# `/src/db`

Local-first SQLite persistence. One Drizzle schema ([schema.ts](./schema.ts)) runs on native
(expo-sqlite) and web (sql.js/WASM). Migrations are generated by drizzle-kit, bundled into
[migrations.generated.ts](./migrations.generated.ts), and applied by `runMigrations`.

## Usage

```ts
import { createDb, seedDemoCards } from '@/db';

const db = await createDb(); // migrations run automatically
seedDemoCards(db);
```

Query functions all take the `db` handle and contain no scheduling/scoring logic (that lives in
`/src/engine`). Import them from `@/db` (e.g. `createCard`, `appendReview`, `upsertAbility`).

## Regenerating migrations after a schema change

```bash
npm run db:migrations   # drizzle-kit generate + bundle into migrations.generated.ts
```

Never hand-edit `migrations.generated.ts`. Never edit an already-shipped migration — add a new
numbered `.sql` file (the bundler picks up files in lexical order).

## Invariants

- **`review_log` is append-only:** no update/delete query functions exist, and DB triggers
  `RAISE(ABORT)` on any UPDATE/DELETE.
- **`fsrs_state` columns equal `CardState` field names** (enum column is `phase`, not `state`), so a
  row maps to a `CardState` with no translation. [fsrs-typecheck.ts](./fsrs-typecheck.ts) guards this
  against ts-fsrs version drift at compile time.
- **`is_synced` / `is_deleted` on `cards` only;** card deletion is soft.
- **`PRAGMA foreign_keys = ON` on both platforms;** **`journal_mode = WAL` on native only** (sql.js
  is in-memory).

## Web durability

Web persists to IndexedDB: loaded on init, debounced-saved after writes, and flushed on
`pagehide`/`visibilitychange`. IndexedDB writes are async, so a flush started on tab-hide usually
completes but is **not guaranteed** — a crash between the last write and the flush can lose that
write. Acceptable pre-sync; revisit when the sync engine lands.

## Verification status

**Verified in CI / on any platform (Node + sql.js):** schema, migrations, the append-only triggers,
and every query function — via `npm test` (`src/db/**/*.test.ts`), plus `npm run typecheck` and
`npm run lint`.

**NOT verified until run on macOS / a browser (the driver is the risk surface).** The schema,
migrations, and queries are shared across platforms, but these depend on the actual driver and stay
unverified until exercised on-device:

- **WAL journal-mode behavior** — `PRAGMA journal_mode = WAL` result and the `-wal` file (native only).
- **expo-sqlite date/boolean coercion** — `timestamp_ms` ⇄ `Date` and `mode:'boolean'` ⇄ 0/1 under
  the expo-sqlite driver (vs sql.js in tests).
- **Transaction semantics** under the expo-sqlite driver.
- **expo-sqlite driver quirks** — bind types, statement finalization, sync vs async.
- **Web runtime** — sql.js wasm loading via Metro (`sql-wasm.wasm` asset) and IndexedDB
  persistence/flush. Blocked from `expo export` verification until an app entry exists.

## iOS smoke-test checklist (run on a Mac / simulator)

Wire a dev-only button to `runDbSelfTest(db)`, then confirm:

- [ ] App boots; `createDb()` resolves with no error.
- [ ] Migrations run once (a second launch does not re-run them — check `_migrations`).
- [ ] `runDbSelfTest` returns `{ ok: true }` (create card → initial state → append review).
- [ ] Data survives a full app restart (the `memory-palace.db-wal` file is present).
- [ ] `PRAGMA journal_mode` reports `wal` on native; the web build never issues WAL.
- [ ] Inserting a `review_log` row for a missing `card_id` fails (FK enforced) — same as web.
